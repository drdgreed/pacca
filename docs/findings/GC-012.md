# Finding: GC-012 — Missing pediatric-complexity escalation branch

**Severity:** SEV-2 (agent-side; healthcare PA correctness defect)
**Surfaced by:** iter-2 chg-2 live baseline run at iter-2 HEAD (2026-05-24)
**Category:** agent-side bug — missing escalation logic (same class as [GC-010](./GC-010.md))
**Recommended fix iteration:** iter-3 (own `chg-` entry, likely bundled with the GC-010 fix)
**Related code:** `src/pacca/agents/clinical_risk_detector.py`, `src/pacca/agents/orchestrator.py`, `src/pacca/agents/decision.py`
**Reproducer:** `python -m tests.clinical.investigate_case GC-012`

## What we observed

GC-012 is *"Pediatric complex case — age + complexity triggers specialist"*.
The case expects `IN_REVIEW` via `branch_2_medical_director` because the
patient is 14 years old and the condition (severe persistent asthma on
maximum ICS/LABA with 3 ED visits) qualifies as "complex." Clinical
eligibility for dupilumab is fully met per NIST/GINA guidelines.

The live pipeline at iter-2 HEAD produces:

| Field | Observed | Expected |
|---|---|---|
| `status` | `AUTO_APPROVED` | `IN_REVIEW` |
| `confidence` | `0.95` | low / N/A (escalation overrides) |
| `correct_outcome` | `False` | `True` |
| `hallucination_detected` | `False` | `False` |
| judge `score` | `2` (consistent across runs) | `≥3` |

The agent correctly notes the patient is 14 (meets the `≥12` age requirement
in NIST/GINA), cites all three required clinical criteria, and approves.
The agent has no awareness of a pediatric-escalation rule.

## Root cause

Same structural gap as GC-010: `branch_2_medical_director` triggers exist in
the harness spec but not in code. Pediatric-complexity specifically is absent
from all three potential enforcement points:

1. `ClinicalRiskDetector.evaluate()` does not check `patient_age` against a
   pediatric threshold combined with a complexity assessment.
2. The `DecisionAgent`'s system prompt does not list age-based escalation as
   a rule it must apply.
3. There is no post-Decision middleware enforcing pediatric review.

Note the `ClinicalCase` model does carry sufficient demographic data to
implement this check; the `Settings` schema already has
`COMPLEXITY_SPECIALIST_REVIEW_MIN` (visible in `.env`), which suggests this
escalation was planned and never implemented.

## Evidence

**Live agent rationale on GC-012** (verbatim, abridged):

> CRITERION 3 - Eosinophilic phenotype: Eosinophil count of 450/uL exceeds
> the guideline threshold of >= 300/uL.
>
> AGE REQUIREMENT: Patient is 14 years old, meeting the >= 12 years age
> requirement.
>
> RECOMMENDATION: APPROVE for dupilumab as add-on therapy for severe
> eosinophilic asthma.

The agent treats age purely as an eligibility criterion (≥12 ✓), not as an
escalation trigger.

**Live judge reasoning on GC-012** (verbatim, abridged):

> The system correctly identified that all clinical criteria for dupilumab
> approval are met […]. However, the system made a critical procedural error
> by AUTO_APPROVING when the expected outcome is IN_REVIEW. The case
> explicitly requires escalation to specialist review due to the patient's
> pediatric age (14 years) combined with moderate-to-high clinical
> complexity […]. The system acknowledged the age requirement but failed to
> recognize that pediatric patients with complex conditions require human
> specialist review as a mandatory escalation trigger.

## Recommended action

Bundle with GC-010 fix into one iter-3 `chg-` entry at
`constraint_level: escalation_branch`. The implementation pattern is
parallel — add `pediatric_complexity_check` alongside `high_cost_check` in
`ClinicalRiskDetector.evaluate()`. Both run after the existing pre-flight
checks; both add new `EscalationReason` enum values; both route to
`branch_2_medical_director`.

**Complexity definition** is the only design decision: when is a pediatric
case "complex enough" to escalate? The `.env` already has
`COMPLEXITY_AUTO_APPROVE_MAX` and `COMPLEXITY_SPECIALIST_REVIEW_MIN`,
suggesting a complexity-score model that doesn't exist yet. Two paths:

- **Conservative (recommended first cut):** any pediatric patient (`< 18`)
  with `severe` or `moderate-to-severe` disease classification routes to
  `branch_2_medical_director`. Simple, defensible, captures the obvious cases.
- **Score-based (eventual):** introduce a `clinical_complexity_score`
  (number of failed prior therapies + presence of comorbidities + disease
  severity tier) and use the `.env` thresholds. Defer until a second
  pediatric case justifies the model.

## Why this is *not* fixed in iter-2

Same charter constraint as GC-010 — iter-2 is "no agent-surface change."
This finding is the input artifact; iter-3 owns the fix.
