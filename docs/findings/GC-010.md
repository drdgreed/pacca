# Finding: GC-010 — Missing high-cost escalation branch

**Severity:** SEV-2 (agent-side; healthcare PA correctness defect)
**Surfaced by:** iter-2 chg-2 live baseline run at iter-2 HEAD (2026-05-24)
**Category:** agent-side bug — missing escalation logic
**Recommended fix iteration:** iter-3 (own `chg-` entry with `escalation_branch` constraint level)
**Related code:** `src/pacca/agents/clinical_risk_detector.py`, `src/pacca/agents/orchestrator.py`, `src/pacca/agents/decision.py`
**Reproducer:** `python -m tests.clinical.investigate_case GC-010`

## What we observed

GC-010 is *"High-cost biologic — criteria met but >$100K annual cost"*. The
case expects `IN_REVIEW` via `branch_2_medical_director` because the
$288,000/year cost exceeds the `HIGH_COST_THRESHOLD` ($100K) configured in
`.env`. All clinical eligibility criteria (ACR 2021 biologic DMARD criteria
for seropositive RA after methotrexate + hydroxychloroquine failure) are met.

The live pipeline at iter-2 HEAD produces:

| Field | Observed | Expected |
|---|---|---|
| `status` | `AUTO_APPROVED` | `IN_REVIEW` |
| `confidence` | `0.98` | low / N/A (escalation overrides confidence) |
| `correct_outcome` | `False` | `True` |
| `hallucination_detected` | `False` | `False` |
| judge `score` | `2` (this run) / `1` (prior baseline run) | `≥3` |

The agent's clinical reasoning is **exemplary** — every ACR criterion cited,
step therapy correctly documented, evidence well-organized, confidence
calibrated appropriately for the clinical case alone. The agent simply has
no awareness of the cost-threshold rule.

## Root cause

The 7-branch escalation tree's `branch_2_medical_director` was designed
(per `docs/HARNESS.md`) to catch business-rule escalations like high cost and
pediatric complexity. In the current codebase, **no component implements
the cost-threshold check**:

1. `ClinicalRiskDetector.evaluate()` runs pre-flight checks (experimental
   procedure codes, rare conditions, prior denials, conflicting guidelines)
   but does **not** check estimated cost against `HIGH_COST_THRESHOLD`.
   Pre-flight returns `should_pre_escalate=False` for GC-010.

2. The `DecisionAgent`'s system prompt
   (`src/pacca/agents/decision_support/system_prompt.md`) lists clinical
   eligibility criteria but does **not** include the high-cost escalation
   rule. The agent cannot consider what it isn't told.

3. There is no post-Decision middleware that compares the request's
   estimated cost against the threshold.

So the cost-trigger branch exists in spec but not in code. The
`HIGH_COST_THRESHOLD=100000` setting in `.env` is read by `Settings` but
never consulted by any code path on the decision route.

## Evidence

**Live agent rationale on GC-010** (verbatim from `investigate_case.py`):

> RECOMMENDATION: APPROVE
>
> This prior authorization request for abatacept (J0129) meets all ACR 2021
> guideline criteria for biologic DMARD therapy in rheumatoid arthritis.
>
> [… complete and correct clinical assessment …]
>
> CONFIDENCE JUSTIFICATION: High confidence (0.98) is appropriate because
> every guideline criterion is explicitly met with specific documented
> evidence.

**Live judge reasoning on GC-010** (verbatim):

> The system correctly identified that all clinical criteria for abatacept
> are met per ACR 2021 guidelines and provided thorough, well-documented
> reasoning on the clinical merits. However, this is a SUBSTANTIALLY WRONG
> decision because it failed to recognize and apply the critical
> cost-threshold escalation rule. The case explicitly states the annual cost
> is $288,000, which exceeds the $100K HIGH_COST_THRESHOLD and should
> trigger IN_REVIEW for Medical Director evaluation regardless of clinical
> approval.

The judge correctly identified the gap. The agent didn't have the rule
available to apply.

**Pattern confirmation via GC-012.** GC-012 (pediatric severe asthma) shows
the same failure mode for a different business-rule trigger
(pediatric+complexity instead of cost). See [GC-012.md](./GC-012.md). Two
cases failing the same way is structural, not incidental.

## Recommended action

Open this as iter-3 work, with its own `chg-` manifest entry at
`constraint_level: escalation_branch`. Two plausible implementations:

**Option A — extend `ClinicalRiskDetector` pre-flight (recommended).**
Add a `high_cost_check` that runs after the existing pre-flight checks,
parses an `estimated_annual_cost` value from the request (currently absent
from the `ClinicalCase` model — needs to be added), and triggers
`EscalationReason.HIGH_COST` if it exceeds `settings.high_cost_threshold`.
Symmetry with the existing pre-flight checks; same failure-path structure.

**Option B — post-Decision middleware.** A middleware hook that runs after
the `DecisionAgent` and rewrites `AUTO_APPROVED → IN_REVIEW` when cost
exceeds the threshold, with a rationale prepended. Cleaner separation of
clinical reasoning from policy enforcement, but adds a fourth tier to the
pipeline that the H1 iter narrative explicitly suggested as future work.

Option A is the smaller change and matches the existing branch_2 design
intent. Recommend it unless iter-3's H2 work (institutional memory) already
plans a middleware-tier introduction.

**What this finding constrains for iter-3's H2 design.** If institutional
memory compresses "RA + abatacept after DMARD failure → approve",
the compression MUST encode the cost guard explicitly:
"approve UNLESS estimated_annual_cost > HIGH_COST_THRESHOLD".
Otherwise H2 would compound the existing escalation gap by encoding it as
a memorized shortcut.

## Why this is *not* fixed in iter-2

iter-2's charter is "no agent-surface change" — pulling forward a fix to
`ClinicalRiskDetector` or `decision_support/system_prompt.md` would
violate it and conflate iter-2's eval-net hardening with a behavioral
bug-fix iteration. The methodology-pure separation is: iter-2 surfaces
the finding; iter-3 fixes the bug as its own attributable change.
